# WatchBuddy 手表独立 MVP 执行计划

> 日期：2026-07-17
> 最终目标：构建并签名可直接安装到 HUAWEI WATCH GT 6 Pro 的 WatchBuddy HAP。
> 日常运行时只有手表端应用活跃，不依赖手机应用、Wear Engine 或其他用户侧设备。

## 1. 不可改变的产品定义

WatchBuddy 必须以手表端独立应用为核心：

- 唯一用户侧应用为 `com.watchbuddy.watch`；
- Android 手机不安装、不启动 WatchBuddy，也不承担计算、传输、语音、传感器或通知职责；
- 不使用 Wear Engine；
- 手表端通过 HTTPS 直接访问 WatchBuddy 服务端；
- 服务端可以承担 AI 推理、主动策略和持久记忆，但用户不需要操作第二台设备；
- 手表离线时允许进入明确的离线模式，不允许静默回退到手机；
- 不安装、不启动、不保活 WatchBuddy 手机应用；手表使用自身或系统提供的网络路径直连服务端。

如果 GT 6 Pro 不向第三方智能穿戴应用开放网络、后台提醒、录音或播放能力，
必须如实记录设备限制。不得为了让测试通过而重新引入手机端兜底。

## 2. MVP 范围

### 必须完成

- 466 × 466 手表角色主页与至少七种可辨识状态；
- 在 WatchBuddy 应用主页内显示可互动的内置宠物，不把应用页面冒充成系统表盘；
- 手表直接连接服务端并获得在线/离线结果；
- 手表发起文字或快捷回复，服务端返回陪伴回应；
- 日常问候、随机关心、关系跟进、每日预算、冷却和安静模式；
- 最近会话、事件、偏好与未完成话题记忆；
- 手表端查看、按条删除和全部清空记忆；
- 网络超时、重复提交、过期消息和离线恢复；
- 构建签名 HAP，并安装到当前 GT 6 Pro 完成真机验收。

### 能力验证通过后纳入

- 从受控服务端目录选择经转换、校验和授权审查的 Codex Pet；
- 在完整性校验、缓存上限和失败回滚均通过后动态更新宠物资源；
- 手表端明确触发的短语音、ASR 和音频播放；
- 手表息屏或应用不在前台时的主动提醒；
- 至少一种来自手表自身的低风险活动或传感器信号。

### 不属于 MVP

- Android 或 iOS 配套应用；
- Wear Engine 手机—手表通信；
- 声称第三方应用可以替换或常驻系统表盘；该能力只能作为独立 Spike 验证；
- 让手表直接加载未经验证的 `pet.json`、远程图片 URL 或原始 Codex 图集；
- 手机通知、手机 TTS、手机麦克风或手机传感器降级；
- 医疗诊断、心理状态断言或未经授权的健康数据上传；
- 将模型密钥直接放进手表应用。

## 3. 目标架构

```text
GT 6 Pro / WatchBuddy HAP
├── 角色 UI 与快捷回复
├── 应用内宠物渲染器
├── 内置手表宠物资源
├── HTTPS API 客户端
├── 设备令牌
├── 最小离线缓存
└── 隐私与记忆控制
        │
        │ TLS / JSON
        ▼
WatchBuddy API
├── 设备注册与鉴权
├── 宠物目录与受控资源清单
├── 会话与幂等
├── companion-core
├── AI 适配器
├── 主动策略调度
└── 记忆存储与删除
```

`packages/companion-core` 中的角色状态、主动策略、记忆模型和协议校验应优先复用到服务端；
旧 Android 代码不得继续作为业务逻辑真源。

必须提前说明具体操作和目的。不得操作用户手机。

记忆。若 GT 6 Pro 的官方调测或系统网络路径需要已配对手机，只允许用户本人完成必要操作；这不等于

- 服务端已提供鉴权后的受控宠物列表、紧凑渲染清单、分页资源摘要与按 ID 下载接口；单帧提供长度、
  ETag 和 SHA-256；当前只发布已审核的 Sprout PNG 轻量包；
- 旧 JS 表端已接入手动触发的宠物目录浏览、Base64 解码、纯 JavaScript SHA-256、PNG 魔数/长度校验、
  临时文件回读、版本化移动和紧凑选择指针；瞬时网络错误最多重试三次，格式/摘要错误不重试；
  73 帧全部成功后才切换，失败或页面隐藏时回滚；
- 动态宠物缓存限制为一个 2MiB 活跃版本，更新过程最多占用 4MiB，新版本提交后清理旧版本；
  目录限制 16 个宠物、单宠物 88 帧，服务端默认单设备每分钟 120 次且表端只串行一个安装任务；
- 466 × 466 浏览器预览已覆盖宠物目录页，标题、说明、容量、状态、导航和底部操作均位于圆形安全区；
- 已提供 codex-pets.net 构建期受控导入器，仅允许官方 HTTPS 下载路径和已识别的可再分发许可证，
  并校验同源重定向、响应/解压预算、ZIP 路径、文件集合、v2 清单及完整 PNG/WebP 图集；站点分享页
  不能单独作为许可证证据，尚待获得一只明确授权的站点 v2 宠物后加入目录；
- 智能穿戴 unsigned HAP 构建已通过；签名、GT 6 Pro 安装、HTTPS、后台、语音与传感器仍需直接证据。

## 9. 宠物资源设计决策

### 9.1 产品边界

第一目标是在用户打开 WatchBuddy 后显示一个缩小版、可互动的宠物。该页面属于第三方应用页面，
不是系统表盘，也不能保证在退出应用、息屏或切换表盘后常驻。系统表盘能力只记录为独立 Spike，
没有官方文档、SDK 工程和真机证据前不得写入产品承诺。

### 9.2 源格式与手表格式

Codex v2 源包的真源是当前 hatch-pet 契约：

- `pet.json` 必须显式包含 `spriteVersionNumber: 2`；
- 图集为 PNG 或 WebP、1536 × 2288、8 列 × 11 行、单格 192 × 208；
- 第 0–8 行为标准动画，第 9–10 行为顺时针 16 个注视方向；
- 第 0 行第 6 列是当前 hatch-pet v2 的中立姿势必需格；
- 其余未使用格必须透明，所有必需格必须非空。

WatchBuddy 不在手表上直接解释这个桌面端格式。构建期转换器先完整校验源包，再生成
`watch-pet.json` 和手表专用小帧。目标尺寸、格式、透明度和帧数必须由 466 × 466 预览、HAP
大小报告与智能穿戴真机验证共同决定，不能先凭经验固定。旧 8 × 9 资源只能进入显式的
兼容迁移分支，不能冒充 v2，也不能阻止 v2 注视方向进入源资产归档。

### 9.3 状态映射

| WatchBuddy 状态或事件 | 首选宠物语义 | 降级规则 |
|---|---|---|
| `idle` | `idle` 循环 | 静态首帧 |
| `sleeping` / `giving_space` | 低速或静态 `idle` | 降低透明度，不持续高频动画 |
| `watching` | `review` | `idle` |
| `daydreaming` / `concerned` | `waiting` | `idle` |
| `curious` | 注视方向或 `jumping` 单次 | `idle` |
| `chatting` / 新消息 | `waving` 单次 | `idle` |
| 请求处理中 | `running` | `waiting` |
| 请求失败或离线 | `failed` 单次 | `idle` 加明确离线文字 |
| 用户轻点 | `jumping` 单次并短振动 | 只振动并同步状态 |

具体时序由 `watch-pet.json` 声明。一次性动作结束后回到服务端角色状态；切页、隐藏和销毁时停止
动画；连续点击只保留一个待播放动作，避免计时器堆积。

### 9.4 转换与安全边界

```text
Codex v2 pet.json + spritesheet
        │
        ├── 来源/许可证/作者审查
        ├── JSON schema、路径和长度校验
        ├── MIME + 魔数 + 像素 + 网格 + 透明度校验
        ├── SHA-256 固定源版本
        ▼
构建期裁帧、缩放、压缩
        │
        ├── 466 × 466 圆屏预览
        ├── 帧/总大小预算
        └── 生成文件摘要
        ▼
watch-pet.json + 受控本地帧
```

服务端导入不得执行包内代码、读取外部路径或信任文件扩展名。远程导入必须限制主机、HTTPS、
重定向、超时、响应字节、解码像素和并发；未知授权、缺少来源或包含第三方商标/角色但无许可的资源
不能进入内置目录。WatchBuddy 原创默认宠物使用项目自有说明和可再分发授权，避免以现有动漫或品牌
角色作为交付资源。

### 9.5 动态同步的降级路径

智能穿戴的 Network Kit 下载、应用沙箱持久文件和运行时图片路径仍待 ArkTS 实现与真机证明。
因此先完成内置默认宠物与构建期多宠物打包。若动态能力不通过，服务器仍可展示可用目录，但用户
选择会进入下一个 HAP 版本，不以手机、Wear Engine 或未经校验的远程图片绕过平台限制。

## 10. 系统表盘 Spike 结论

华为将“智能穿戴应用”和“表盘主题”定义为两条独立交付路线。系统表盘需要使用
Theme Studio Pro 创建 466 × 466 表盘工程，同时处理常亮与息屏/AOD 状态，再单独导出、测试和发布；
它不是 `apps/watch-huawei-wearable` 生成的智能穿戴 HAP。

官方主题文档给出了序列帧图片和按钮触摸能力，因此可以继续验证“打包进表盘的简化宠物动画与轻点
反馈”。但当前没有官方证据证明表盘主题能运行 WatchBuddy JS 应用、直接调用任意 HTTPS API、
读取 HAP 动态下载的宠物文件，或承载 AI 对话和记忆功能。因此产品边界固定为：

- 完整宠物互动、联网对话、记忆和在线换宠物继续运行在 WatchBuddy 应用内；
- 系统表盘只作为后续独立主题产物，第一版最多使用随表盘打包的静态/序列帧宠物；
- Theme Studio Pro 工程、GT 6 Pro 导入与真机交互未完成前，不对外宣称系统表盘可用；
- 表盘路线失败时不回退到手机、Wear Engine、悬浮窗或伪装常驻。
